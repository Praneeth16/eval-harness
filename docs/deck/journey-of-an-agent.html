04 · 32 · Part 01Journey of an agent
Why bother

Evals are broken. Use them anyway.

Every objection is true: evals are noisy, subjective, gameable, and never complete. None of that is a reason to skip them.

  • Skipping evals does not remove the failures. It just moves discovery to your customer.
  • The bar is not "a benchmark you can trust." The bar is "would this have caught last week's incident?"
  • An imperfect eval that catches the cold-open failure beats a perfect one you never wrote.

Treat the suite as a memory of failures you refuse to repeat, not a leaderboard. Then a broken-but-improving eval compounds.

The distinction everything rests on

An agent's output is not its outcome.

Output
The text it returned

"The restaurant is booked." A string. Cheap to read, cheap to grade, and easy to fake.

≠
Outcome
The state of the world after

Is there actually a reservation? For any agent that acts, scoring the output is a category error.

Score the outcome, and the path that produced it. That means instrumenting the agent, not just reading its answer.

Who does the grading

Three kinds of grader. Use all three.

Human

The ground truth

The definitive standard. Slow and expensive, but the only source of real taste.

trustworthy · does not scale
Code-based

Deterministic checks

String match, assertions, test execution, tool-call verification in the transcript.

cheap, reproducible · misses semantics
Model-based

LLM as judge

Scores subjective quality at scale. Fast and flexible, with systematic biases.

scales · must be calibrated

Human for calibration, code for constraints, model for scale. No single grader is enough.

10 · 32 · Part 01Journey of an agent
The grader you cannot skip

Human eval is the calibration baseline.

Every automated grader is only as trustworthy as the human judgment it was checked against.

Humans rarely agree without a refined rubric. So the work is not "have a human look," it is building the annotation guide that makes two reviewers agree, then holding your judges to it.

How it matures
  • Start: manual inspection, vibe checks.
  • Then: a written rubric, pass / fail or Likert.
  • Keep a 50-example "gold gold" set, labeled by hand.
  • Re-measure judge-vs-human agreement on every prompt change.

Humans do not grade every run. They grade the graders.

11 · 32 · Part 01Journey of an agent
Model-based grading, up close

A judge is a cheap reviewer with systematic biases.

Setup 1

Pairwise

Judge picks the better of two responses.

Setup 2

Direct

Judge assigns one response a score.

Setup 3

Reference-guided

Judge gets a gold reference plus the candidate.

  • Known biases: prefers the first response, prefers longer answers, prefers its own style.
  • A single 1 to 5 score is almost meaningless. Decompose into many binary, itemized questions.
  • Judges agree with each other more than with humans. Calibrate against the gold set, ship above a fixed agreement floor.
12 · 32 · Part 01Journey of an agent
Grading the path, not just the answer

Five ways to grade what the agent did.

invocation
Did it decide to call a tool, or not, correctly?
selection
Did it call the right tool?
structural
Was the call well-formed: right args, right shape?
trajectory
Was the sequence of calls right, in the right order?
outcome
Did the run achieve the result, whatever the path?

Grade at least two. We lean on trajectory and outcome: the answer can be right for the wrong reason, and only the path tells you which.

Case study · the reliability benchmark

A 70% agent is not 70% reliable.

attempts (k) → success rate → pass@k · solved at least once pass^k · solved every time
Benchmarks like τ-bench separate two questions.

pass@k: can it ever succeed? pass^k: does it succeed every time? For customer-facing agents, only the second one matters, and it collapses as k grows.

τ-bench: multi-turn tool-agent tasks in real domains, scored on consistency, not best-of-N.

The agent under test

Quill answers security questionnaires.

A LangGraph pipeline that parses the questionnaire into atomic questions, retrieves over a 30-document policy corpus, drafts cited answers, and flags gaps. Every node emits an MLflow span. We grade it on a 20-question golden set.

Parse + classify

Split the questionnaire into atomic questions. Route each: security, compliance, commercial, technical.

Retrieve + draft

FAISS over policies, past responses, framework controls. Draft answers grounded in citations.

Verify + tier

Check every citation against the corpus. Flag gaps with no policy. Risk-tier for human review.

Quill is just the example. The harness is the reusable part: swap in any agent that retrieves, cites, and acts.

The Swiss cheese strategy

No scorer is complete. Each one fails differently.

Runs first

Deterministic

Regex, format, citation prefix, latency and cost budgets. Cheap, fast.
misses: semantic errors
Runs second

Trajectory

Walk the span tree. Did verify fire before the citation was written?
misses: content errors
Runs third

LLM judge

Reviewer-accept, relevance, adherence. The only layer that costs tokens.
misses: structural errors
Runs fourth

Safety

Overclaim detection, prefix filters, refusal correctness on the adversarial set.
misses: nothing it is told to find

A scorer with a bug is worse than no scorer: it produces false confidence. Each layer is named for the exact failure it catches.

20 · 32 · Part 03Journey of an agent
The shape of the cold-open bug

One call to write the answer and the citations.

def drafter(state):
  response = llm.generate(prompt_with_retrieval(state.question, state.chunks))
  return {"answer": response.text, "citations": response.citations}

The citations are whatever the model decided to attach. Retrieval ran earlier, the verifier runs later. The answer was already written before either happened.

It passes deterministic scorers (well-formed strings) and the judge (relevant answers). It fails exactly like the cold open: the evidence is in the ordering, never in the string.

21 · 32 · Part 03Journey of an agent
The fix

Propose, verify, then finalize.

Step 1

Propose

The model emits candidate citations as JSON. It does not yet write prose.

Step 2

Verify

A deterministic tool checks every candidate against the corpus. Phantom citations die here.

policy_exists_check()
Step 3

Finalize

The model writes the answer from the verified list only. A post-check matches every citation against it.

Three claims, all provable from the span tree: verify fired before finalize, every cited ID is in the verified list, no citation appears that was never proposed. Any one alone is theater. All three is honesty.

The model is creative but inconsistent. The verifier is rigid but reliable. Hand the verifier to the model and you get creativity bounded by correctness.

If you cannot write a verifier for your domain, you are doing harder work than you think. Most teams discover this the first time they try.

27 · 32 · Part 04Journey of an agent
Step 04 of the flywheel, up close

GEPA reads your failures and rewrites the prompt.

What it is
  • An optimizer that reads failing traces, writes a text diagnosis, and proposes a new prompt. Then repeats.
  • Genetic + Pareto: it keeps a population of prompts and selects on many objectives at once.
  • The gradient is plain language: the feedback string is what the reflection model reads.
How it helps
  • Explores prompt variants a human would skip as awkward or redundant.
  • Every candidate is graded by your scorers before it survives a round.
  • Multi-objective: you keep a frontier of tradeoffs, not one fragile winner.

It does not invent ground truth. It searches the space your scorers can already rank, which is why the scorers have to be honest first.

28 · 32 · Part 04Journey of an agent
DSPy + GEPA · the loop

If scorers can rank prompts, let the system write them.

01

Failed traces

Run the metric. Collect concrete CLEAR-S failures: phantom IDs, overclaims, unverified refs.

02

Reflect

A stronger LM reads the feedback string and diagnoses what went wrong, in text.

03

Mutate

It proposes a prompt patch. The feedback string is the gradient the reflection LM reads.

04

Pareto select

The candidate is graded on all 7 axes. Survives only if non-dominated on some combination.

repeat · a few hundred metric calls is the floor where reflective mutation beats noise

A human edit is a hypothesis. A GEPA mutation is a hypothesis and an experiment, graded before it survives.

Five axioms.

Trace before you eval. You cannot grade logic you cannot see.
02
Eval layers stack: deterministic for constraints, judge for tone, trajectory for logic.
03
Static prompts ship hallucinations. A self-evolving harness compounds away from them.

We built CI for code. Agents need CI for behavior. This harness is that CI.
