## Creating AI-Generated Opportunity Solution Trees

Two services built for Vistaly:
1. AI-generated interview snapshots
2. AI-generated opportunity solution trees

Design partners upload three customer interviews; the system identifies key moments and generates a complete tree from snapshots.

---

## Integrating New Interviews into an Existing Tree

Updating a tree with new interview content proved significantly more challenging than generating from scratch.

### The Shared Philosophy

Both teams share a core principle: they explicitly do not want users to treat AI-generated trees as factual. Instead, the goal is to "engage with them, correct them, collaborate with the AI." Scaffolding cross-interview synthesis rather than automating it entirely.

### The Git Diff Approach

Drew inspiration from git diff—an equivalent for opportunity solution trees showing users step-by-step what would change when new interviews were added.

---

## The Prototype-to-Production Gap

"It's fairly easy to get to an impressive prototype with AI, but much harder to get to a real production-quality product."

### Data Preservation Challenges

Each tree node contains:
- **Source opportunities:** interview opportunities providing supporting evidence
- **Children:** tree sub-opportunities

When updating through merges, splits, or deletes, all data must be preserved.

### Early Success and Hidden Complexity

Around the 14th-16th interview uploads, the system began exhibiting unexpected behavior. Change sets (step-by-step instructions) generated by the service were not always producing the final tree the service had output. This violated the core data contract.

---

## Tree Diffs Are Ambiguous

The fundamental problem: tree diffs—unlike linear document diffs—are inherently ambiguous.

### The Split Operation

Opportunity A (sources 1,2,3; children x,y,z) splits into A and B:
- A gets sources 1,2 and child x
- B gets source 3 and children y,z

The model must decide how to distribute.

### The Merge Operation

When opportunity B (source 3, children y,z) merges with C (sources 4,5, children t,u,v):
- C absorbs all sources: 3,4,5
- C absorbs all children: t,u,v,y,z

### The Hidden Moves Problem

Diffing input vs output trees obscures the split and merge operations. "It looks to our diff like some of A's sources and children moved from A to C. But we don't know why. The split and the merge conceptually explain what happened, but those moves are invisible to the diff."

### The Semantic Problem

The team needed to show *why* it changed so users could evaluate the changes—reconstructing each move step by step.

---

## Asking the Model to Show Its Work

Refactored all prompts to require the model to generate both the final output and the step-by-step moves using predefined change set language.

### Two Error Categories

1. **Invalid moves:** Action not permitted by the rules (e.g., merging A into B when A has children—orphaning them)
2. **Discrepancies:** Change set did not generate the recommended output tree

Reduced discrepancy rate to ~1 in 40 calls. With 10-20 LLM calls per service run, error appeared in ~50% of runs—unacceptable.

---

## The Insight: Let the Model Correct Its Own Mistakes

Validation code became a tool for an agentic loop:

1. Model generates change set
2. Validation tool checks if change set produces the output tree
3. If validation passes, loop exits
4. If validation fails, results sent back to model to correct

### Production Implementation

**Workflow:**
1. Initial model call handles the tree-building task
2. Model outputs a tool request (change set)
3. Loop runs validation code
4. If validation passes, tool emits success signal and loop exits
5. If validation fails, tool returns specific fix instructions
6. Model attempts corrections and calls validation tool again
7. Loop continues until validation passes or max turns reached

"Errors are still theoretically possible. The repair loop can fail and hit max turns. But I haven't seen it in all of my testing. The agent typically fixes the bad change set in one to two turns."

## Central Argument

While AI prototypes can be built quickly, production-quality AI products require solving legitimately hard technical problems. Tree-diff ambiguity, compound operations, and data preservation requirements created unexpected complexity beyond the initial prototype.

The solution—using the model itself as an agent capable of validating and correcting its own work—illustrates how AI engineers build reliable systems by combining LLM capabilities with deterministic validation tools in iterative loops.

The deterministic-verifier-in-the-loop pattern: LLMs are creative but inconsistent; deterministic code is rigid but reliable. Hand the verifier to the LLM as a tool, and you get creativity bounded by correctness.
